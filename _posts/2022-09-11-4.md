`GraphQL`은 2012년 페이스북에서 개발해서 2015년에 공개적으로 발표했습니다. 여러 개발자들 사이에서 자주 입에 오르내리고 있고, 한번 사용해보면 `REST API`를 더 이상 사용하지 않게된다는 등의 좋은 평가가 많아 알게되었습니다. 그러다가 외부 회사에서 개발되어있던 웹 프로젝트 소스를 이어받아 유지보수 업무를 받게되었는데 `GraphQL`이 사용되고 있었던겁니다. 반 강제적으로 `GraphQL`을 사용해야하는 상황이 된거죠. 그래서 저는 GraphQL에 대한 간단한 소개와 사용 후기를 정리해볼까합니다.

## 쿼리 언어

`GraphQL`은 쿼리 언어입니다. 클라이언트에서 서버에 필요한 데이터를 요청할때 데이터의 구조를 지정해서 받아온다는 것이 특징입니다. 쿼리 언어라고 하면 흔히 `SQL`을 떠올리게 되는데요, 그러나 일반적으로 서버에서 데이터베이스로 요청을 보내는 `SQL`과 달리 `GraphQL`은 클라이언트에서 서버로 보내는 요청에 대한 규격입니다. 그렇기에 `SQL`과 같은 데이터베이스 쿼리 보다는 `REST API`와 비교하는 것이 더 바람직합니다.

## REST API 와의 차이

사실 `GraphQL`과 `REST API`의 네트워크 요청 방식에는 차이가 없습니다. 실제로 클라이언트에서 보내는 `GraphQL` 요청을 개발자 도구로 확인해보면 POST 요청으로 나오는 것을 확인할 수 있습니다. 다만 개발의 구조에서 차이가 나게됩니다.

### 모든 요청을 하나의 Endpoint로

목적과 기능에 따라 매번 새로운 Endpoint를 생성하는 `REST API`와 달리 `GraphQL`은 하나의 Endpoint 만을 갖습니다. 어떠한 기능을 수행할지는 클라이언트에서 보내는 쿼리에 따라 달라집니다 (일반적으로 Query 와 Mutation 방식으로 나뉩니다).

### 받아오는 데이터 구조를 지정

요청에 대한 응답의 구조를 서버에서 지정한대로만 내어주는 `REST API`와 다르게 클라이언트에서 지정한대로 서버에서 구조를 맞춰서 가져다줍니다.

## 사용 후기

기존에 사용하던 `REST API`와 비교했을때 장단점이 뚜렷하다고 느껴집니다.

### 장점

- Overfetching 문제 해결 및 요청수 최적화에 적합
- 클라이언트에서 데이터 구조에 대한 제어가 가능
- 확장성에 용이

### 단점

- 높은 진입장벽
- 클라이언트 개발자에게 데이터 구조에 대한 이해를 강제
- 장점을 잘 활용하지 못하는 경우 안쓰는 것만 못함
- 파일 전송과 같은 특수한 요청
